One task is one outcome confirmable by a single check — a candidate that needs two separate checks to call it done is two tasks. `T#` is a permanent anchor, assigned once and never renumbered — the same number a later execution log entry cites.

<!-- role:dispatcher -->
## Metering

Hand the planner the committed Approach (plus relevant research, if any); assign each returned candidate the next `T#` in sequence and meter them into `plan.md` a few per pass, keeping the document legible. A task once numbered keeps its number even if later tasks are added ahead of it in dependency order.
<!-- /role -->

<!-- role:worker -->
## Decomposing

Decompose the Approach into the smallest set of tasks that covers it, each satisfying the one-outcome test. Order by dependency, not guessed priority — each task lists `Depends on: none` or its prerequisites by title. Propose candidates only: never assign permanent numbers and never write `plan.md` — numbering and metering are the skill's job.
<!-- /role -->
